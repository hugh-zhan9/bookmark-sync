# 白屏卡死（IPC 互斥锁重入）设计说明

> **目的**：仅修复白屏/卡死问题，不引入新功能。

## 背景与根因
- 启动时默认执行 `sync_event_pull_only`（事件同步启动开关默认 true）。
- `sync_event_pull_only` / `sync_event_push_only` / `sync_github_incremental` 内部同线程二次 `state.router.lock()`，触发不可重入的 `std::sync::Mutex` 死锁。
- IPC 请求进入后端阻塞在锁等待，导致 UI 线程卡死并表现为白屏。

## 目标与范围
- 目标：消除互斥锁重入导致的死锁，保证启动与同步流程可用。
- 范围：只调整锁的作用域与使用顺序，不改业务逻辑，不改 UI。
- 不做：不加超时提示，不做降级模式，不改配置结构。

## 方案（锁策略）
- 在同步相关命令中：
  - 仅在读取 `router.kind()` 与获取 `sqlite` 连接句柄时短暂持有一次 `router` 锁。
  - 立刻释放锁，后续不再嵌套 `state.router.lock()`。
- 保持原有错误处理与返回值，不改变功能语义。

## 影响评估
- 行为不变，仅消除死锁。
- 预期影响：启动时 IPC 不再阻塞，白屏/卡死消失。

## 测试策略
- 新增单元测试：验证在持有 `router` 锁时可通过辅助函数安全获取 sqlite 连接与数据源类型（不触发二次锁）。
- 本地回归：`npm test` + `cargo test`，并手动启动应用验证启动不卡死。
